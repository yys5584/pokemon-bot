## 1. 개요

### 1.1 프로젝트 정의

텔레그램 미니앱(Telegram Mini App) 기반의 **3D 배틀 시각화 시스템**으로, 토너먼트 결승전에서 포켓몬 배틀을 실시간 3D 연출로 제공한다. 기존 텍스트 기반 배틀 결과를 시네마틱한 3D 애니메이션으로 변환하여 유저에게 몰입감 있는 관전 경험을 선사한다.

### 1.2 핵심 가치

| 항목 | 설명 |
|------|------|
| **시각적 임팩트** | Three.js WebGL 기반 풀 3D 배틀 연출 |
| **접근성** | 텔레그램 내 별도 설치 없이 즉시 실행 |
| **단일 파일 배포** | HTML 단일 파일로 CDN 배포, 유지보수 용이 |
| **커뮤니티 참여** | 토너먼트 참가자가 관중으로 등장하는 소속감 |

### 1.3 적용 범위

- **1차 적용**: 토너먼트 결승전 (2인 대결)
- **2차 확장 예정**: 랭크전 명예의 전당, 프리미엄 유저 일반 배틀
- **장기 목표**: 전 배틀에 경량 3D 연출 옵션 제공

## 2. 핵심 기능 상세

### 2.1 3D 배틀 연출

#### 2.1.1 포켓몬 3D 모델 로딩

GitHub에 호스팅된 GLB(glTF Binary) 포맷의 포켓몬 3D 모델을 동적으로 로드한다.

**모델 소스 구조:**
```
https://raw.githubusercontent.com/{repo}/models/
├── normal/
│   ├── 001.glb    # 이상해씨
│   ├── 006.glb    # 리자몽
│   ├── 150.glb    # 뮤츠
│   └── ...
└── shiny/
    ├── 006.glb    # 이로치 리자몽 (골드 텍스처)
    └── ...
```

**로딩 파이프라인:**
```
모델 URL 결정 → GLTFLoader 로드 → DRACOLoader 압축 해제
→ 텍스처/머티리얼 적용 → 애니메이션 믹서 연결 → 씬 배치
```

**모델 규격:**
| 항목 | 기준값 | 비고 |
|------|--------|------|
| 폴리곤 수 | 5,000~15,000 tri | 모바일 최적화 |
| 텍스처 해상도 | 512x512 | PNG/WebP |
| 파일 크기 | 500KB~2MB | DRACO 압축 적용 |
| 애니메이션 | idle, attack, hit, faint | 최소 4종 |
| 원점 | 발밑 중앙 | Y-up 좌표계 |

**이로치(샤이니) 처리:**
- 별도 shiny 모델이 있으면 우선 사용
- 없으면 기본 모델에 셰이더 오버레이 적용 (반짝임 + 색조 변환)
- 이로치 등장 시 추가 파티클 이펙트 (별 모양 스파클)

#### 2.1.2 타입별 스킬 이펙트

7가지 주요 타입에 대해 각각 고유한 시각 이펙트를 구현한다.

##### 불꽃 (fire) — `화염방사`, `불대문자`
```
[발사]                    [피격]
파티클: 주황~빨강 그라데이션   폭발: 방사형 화염 버스트
형태: 원뿔형 스트림           잔여: 바닥 화염 + 연기
크기: 반경 0.3 → 1.5        지속: 0.8초
속도: 15 units/sec          화면: 주황 틴트 오버레이
사운드: 불꽃 분사음 →         사운드: 폭발음 + 지글거림
```

##### 대화염 (fire_blast) — `불꽃세례`, `오버히트`
```
[발사]                    [피격]
파티클: 대(大)자 형태 화염    폭발: 화면 전체 화염 커버
형태: 전방 확산형             잔여: 3초간 바닥 불기둥
크기: 반경 2.0 → 4.0        지속: 1.5초
속도: 10 units/sec          화면: 빨강 + 왜곡(heat haze)
특수: 발사 시 지면 갈라짐     사운드: 대폭발 + 여진
```

##### 사이코 (psychic) — `사이코키네시스`, `미래예지`
```
[발사]                    [피격]
파티클: 보라~분홍 에너지 링   폭발: 구형 사이코 웨이브
형태: 동심원 확장             잔여: 부유하는 잔상 파편
크기: 반경 0.5 → 3.0        지속: 1.0초
속도: 12 units/sec          화면: 보라 틴트 + 크롬 수차
특수: 대상 포켓몬 공중 부양   사운드: 고주파 울림 → 폭쇄음
```

##### 어둠 (dark) — `악의파동`, `섀도볼`
```
[발사]                    [피격]
파티클: 검보라 어둠 구체      폭발: 블랙홀형 흡입 → 폭발
형태: 불규칙 맥동 구체        잔여: 바닥에 검은 안개
크기: 반경 0.8 → 1.2        지속: 1.2초
속도: 20 units/sec          화면: 비네팅 강화 + 명도 저하
특수: 궤적에 그림자 잔상      사운드: 저음 웅웅거림 → 파열음
```

##### 바람 (wind) — `에어슬래시`, `폭풍`
```
[발사]                    [피격]
파티클: 반투명 바람 블레이드   폭발: 토네이도형 상승기류
형태: 초승달 슬래시 x3        잔여: 나뭇잎/먼지 파티클
크기: 폭 2.0, 두께 0.1      지속: 0.7초
속도: 25 units/sec          화면: 모션블러 강화
특수: 카메라 미세 흔들림      사운드: 바람 절단음 → 윈드벌스트
```

##### 얼음 (ice) — `냉동빔`, `눈보라`
```
[발사]                    [피격]
파티클: 하늘~흰 얼음 결정     폭발: 방사형 얼음 기둥 솟구침
형태: 직선 빔 + 결정 나선     잔여: 피격 지점 얼음 코팅
크기: 반경 0.2 → 0.5        지속: 1.0초
속도: 18 units/sec          화면: 서리 오버레이 + 채도 저하
특수: 궤적 위 눈 결정 잔존    사운드: 결정화음 → 파쇄음
```

##### 빔 (beam) — `파괴광선`, `하이퍼빔`
```
[발사]                    [피격]
파티클: 금~흰 에너지 집중     폭발: 대규모 에너지 방출
형태: 차징(0.5초) → 직선 빔   잔여: 지면 함몰 + 잔불
크기: 반경 0.3 → 2.0        지속: 2.0초 (최장)
속도: 즉발 (빔 확장)         화면: 화이트아웃 → 복귀
특수: 차징 중 카메라 줌인     사운드: 차징음 → 빔 발사 → 대폭발
```

**이펙트 공통 사양:**
| 파라미터 | 기본값 | 설명 |
|----------|--------|------|
| 파티클 수 | 200~500 | 타입별 조정, 저사양 모드 50% 감소 |
| 수명 | 0.3~1.0초 | 개별 파티클 수명 |
| 블렌딩 | AdditiveBlending | 빛나는 이펙트 연출 |
| 감쇠 | 크기 0→max→0 | 이즈인-아웃 커브 |

#### 2.1.3 시네마틱 카메라 워크

각 턴마다 4단계 카메라 시퀀스를 실행한다.

```
턴 시작
  │
  ▼
[Phase 1] 공격자 줌인 (0.6초)
  │  카메라: 전체 뷰 → 공격자 우측 45° 클로즈업
  │  연출: 배경 DOF(피사계심도) 블러 강화
  │  의도: "이 포켓몬이 공격한다" 시각적 신호
  │
  ▼
[Phase 2] 스킬 이펙트 (0.8~2.0초, 타입별)
  │  카메라: 공격자 → 피격자 방향으로 패닝
  │  연출: 이펙트 궤적 추적, 속도감 있는 이동
  │  의도: 스킬의 위력과 속도감 전달
  │
  ▼
[Phase 3] 피격 줌 (0.5초)
  │  카메라: 피격자 정면 클로즈업 (약간 아래에서 위로)
  │  연출: 피격 리액션 + HP 바 감소 애니메이션
  │  의도: 데미지의 심각도를 시각적으로 전달
  │
  ▼
[Phase 4] 복귀 (0.8초)
  │  카메라: 전체 뷰로 부드럽게 복귀
  │  연출: DOF 해제, UI 요소 페이드인
  │  의도: 전체 전황 파악 + 다음 턴 준비
  │
  ▼
다음 턴 or 배틀 종료
```

**카메라 포지션 프리셋:**
```

              관중석 (북쪽)

    │  👥👥👥👥👥👥👥👥👥👥👥👥  │
    │  👥👥👥👥👥👥👥👥👥👥👥👥  │
```

#### 2.1.4 타격감 시스템

배틀의 임팩트를 극대화하는 6가지 타격감 요소를 중첩 적용한다.

##### 히트스톱 (Hit Stop)
```
피격 순간 → 전체 씬 0.08초 정지 (Time.timeScale = 0)
→ 정지 해제 후 1.2배속으로 복귀 (보상 가속)
```
- 크리티컬 히트: 0.15초로 연장
- 효과: 타격 순간의 무게감 극대화

##### 쇼크웨이브 (Shockwave)
```
피격 지점에서 동심원 확장 (반경 0~5, 0.4초)
셰이더: 굴절(refraction) 효과로 공간 왜곡
```
- 크리티컬: 웨이브 2중 발생
- 효과적 타입: 웨이브 색상이 타입 컬러로 변경

##### 파편 (Debris)
```
피격 시 지면에서 돌/흙 파편 3~8개 튀어오름
RigidBody 시뮬레이션 (간이 물리)
크기: 0.1~0.3 유닛
수명: 1.5초 (페이드아웃)
```

##### 모델 플래시 (Model Flash)
```
피격 포켓몬 모델 → 0.05초 흰색 → 원래 색 → 0.05초 흰색 → 원래 색
총 3회 플래시 (0.3초)
구현: MeshBasicMaterial(white) 교체 → 복원
```

##### 화면 흔들림 (Screen Shake)
```
카메라 오프셋: random(-intensity, intensity) per frame
일반 공격: intensity = 0.1, duration = 0.2초
크리티컬: intensity = 0.3, duration = 0.4초
감쇠: 선형 감소
```

##### 블룸 스파이크 (Bloom Spike)
```
피격 순간: UnrealBloomPass strength 1.0 → 3.0 (0.1초)
→ 3.0 → 1.0 (0.3초, ease-out)
크리티컬: strength 최대 5.0
```

### 2.3 트레이너 시스템

#### 2.3.1 개요

포켓몬 뒤에서 트레이너가 지시하고 응원하는 모습을 표현하여, 유저(트레이너)와 포켓몬의 유대감을 시각화한다.

#### 2.3.2 트레이너 모델 디자인

**기본 트레이너 모델 사양:**
```
┌──────────┐
│  👤 머리  │  ← 프로필 사진 매핑 영역 (원형 마스크)
├──────────┤
│          │
│  몸통    │  ← 트레이너 코트 (팀 컬러 적용)
│          │
├──────────┤
│  하체    │  ← 바지 + 신발
└──────────┘

높이: 1.8 유닛 (포켓몬 대비 인간 비율)
폴리곤: 3,000~5,000 tri
스타일: 세미 리얼 (포켓몬 세계관 톤)
```

**트레이너 외형 변형:**
| 요소 | 기본 | 프리미엄 구독자 | 채널장 구독자 |
|------|------|----------------|--------------|
| 코트 | 회색 기본 코트 | 금테 장식 코트 | 챔피언 망토 |
| 모자 | 기본 캡 | 골든 캡 | 왕관 |
| 이펙트 | 없음 | 발밑 미세 파티클 | 오라 이펙트 |
| 입장 | 기본 등장 | 빛줄기 입장 | 번개 입장 |

#### 2.3.3 프로필 사진 얼굴 매핑

유저의 텔레그램 프로필 사진을 트레이너 모델의 얼굴 영역에 매핑한다.

**처리 파이프라인:**
```
텔레그램 프로필 사진 URL
        │
        ▼
  이미지 다운로드 (CORS proxy 경유)
        │
        ▼
  Canvas에서 원형 마스크 + 리사이즈 (128x128)
        │
        ▼
  THREE.CanvasTexture 생성
        │
        ▼
  트레이너 머리 메시의 머티리얼에 적용
        │
        ▼
  빌보드 방식으로 항상 카메라를 향하게 처리
```

**대안 (프로필 사진 없을 때):**
- 기본 실루엣 + 닉네임 첫 글자 표시
- 밝은 색 원형 배경에 이니셜

#### 2.3.4 트레이너 애니메이션

| 애니메이션 | 트리거 | 설명 | 길이 |
|-----------|--------|------|------|
| `idle` | 기본 상태 | 양손 주머니, 미세 체중이동 | 2초 루프 |
| `command` | 자기 턴 공격 시 | 한 손 전방 지시 포인트 | 0.8초 |
| `throw_ball` | 포켓몬 교체 시 | 오버헤드 투척 모션 | 1.2초 |
| `cheer` | 크리티컬/효과적 공격 | 주먹 쥐고 환호 | 1.0초 |
| `worry` | 자기 포켓몬 HP 25% 이하 | 한 발 앞으로 + 걱정 제스처 | 1.5초 루프 |
| `victory` | 승리 시 | 점프 + 양팔 만세 | 2.0초 |
| `defeat` | 패배 시 | 고개 숙임 + 한 손 모자 | 1.5초 |

### 2.4 관중 시스템

#### 2.4.1 개요

토너먼트에 `ㄷ`을 눌러 참가한 유저들이 관중석에 표현된다. 단순한 배경이 아니라, 배틀 상황에 따라 동적으로 반응하는 살아있는 관중을 구현한다.

#### 2.4.2 관중 수 스케일링

| 실제 참가자 수 | 표시 관중 수 | 관중석 규모 | 분위기 |
|---------------|-------------|------------|--------|
| 1~10명 | 실제 수 x1 | 1단 소형 | 아담한 동네 대회 |
| 11~30명 | 실제 수 x1 + NPC 10명 | 2단 중형 | 활기찬 지역 대회 |
| 31~60명 | 실제 수 x1 + NPC 20명 | 3단 대형 | 열광적 전국 대회 |
| 61~100명 | 실제 수 x1 + NPC 40명 | 3단+VIP석 | 국제 토너먼트 |
| 100명+ | 100명 + 나머지 NPC 채움 | 풀 스타디움 | 월드 챔피언십 |

> **NPC 관중**: 참가자가 아닌 일반 모델. 분위기 채우기 용도.
> **실제 참가자**: 닉네임 표시 가능, 우선 배치 (앞줄, 중앙).

#### 2.4.3 관중 모델

```
[관중 개별 모델 - 로우폴리]

   ○     ← 머리 (구체, 피부색 랜덤)
  /|\    ← 상체 (색상 랜덤 티셔츠)
  / \    ← 하체 (간략화)

폴리곤: 100~200 tri (초경량)
높이: 0.8 유닛
바리에이션: 8가지 컬러 조합
```

**참가자 관중 (하이라이트):**
- 일반 관중 대비 약간 밝은 조명
- 머리 위 닉네임 태그 (간헐적 표시)
- 프로필 사진이 있으면 머리에 텍스처 매핑

#### 2.4.4 관중 반응 애니메이션

| 반응 | 트리거 조건 | 애니메이션 | 참여율 | 사운드 |
|------|------------|-----------|--------|--------|
| **기본** | 상시 | 좌우 미세 흔들림 | 100% | crowd_idle |
| **박수** | 매 턴 종료 | 양손 박수 | 40~60% | 박수 소리 |
| **환호** | 크리티컬 히트 | 양팔 들기 + 점프 | 70% | crowd_cheer |
| **놀람** | 효과적 공격 | 양손 입 가리기 + 뒤로 젖히기 | 80% | crowd_gasp |
| **기립박수** | KO / 결승 마지막 턴 | 전원 기립 + 박수 | 95% | crowd_standing |
| **웨이브** | 특수 연출 (빔 타입) | 좌→우 순차 기립 | 100% (순차) | 웨이브 환호 |
| **한숨** | 포켓몬 KO (해당 쪽 팬) | 머리 감싸기 | 30% | 한숨 |

> **참여율**: 해당 반응을 실행하는 관중의 비율. 100%가 아닌 경우 랜덤 선택 + 0.1~0.3초 시차로 자연스러움 연출.

#### 2.4.5 닉네임 표시 연출

관중 중 실제 참가자의 닉네임이 간헐적으로 표시되어, "나도 저기 있다"는 소속감을 준다.

```
[닉네임 표시 규칙]

1. 매 턴 시작 시 랜덤 2~4명의 닉네임 말풍선 표시
2. 표시 시간: 1.5초 (페이드인 0.3초 → 유지 0.9초 → 페이드아웃 0.3초)
3. 크리티컬/KO 시 해당 쪽 응원 관중 5~8명 동시 표시
4. 결승 승리 순간: 전체 참가자 닉네임 동시 표시 (3초)

[말풍선 스타일]
┌──────────┐
│  문유     │  ← 둥근 모서리, 반투명 배경
└────┬─────┘
     ▽
    👤         ← 해당 관중 위에 배치
```

### 2.5 몬스터볼 연출

#### 2.5.1 개요

포켓몬 교체 시 몬스터볼 투척과 오픈 애니메이션을 통해 교체 과정을 극적으로 연출한다.

#### 2.5.2 몬스터볼 3D 모델

```
[몬스터볼 모델 사양]

    ╭──────╮
   ╱ 빨강   ╲     ← 상반구: 빨강 (PBR 글로시)
  │──●────── │     ← 중앙 띠: 검정 + 버튼(흰 원)
   ╲ 흰색   ╱     ← 하반구: 흰색 (PBR 글로시)
    ╰──────╯

폴리곤: 500 tri
지름: 0.3 유닛
텍스처: 512x512 (상반구 빨강, 하반구 흰, 중앙 검정 띠)
반사: 환경 맵 반사 (메탈릭 0.3, 러프니스 0.4)
```

### 4.2 데이터 전달 방식

**방법 1: URL 쿼리 파라미터 (소규모 데이터)**
```
https://tgpoke.com/battle?data={base64_encoded_json}
```
- 장점: 단순, 별도 API 불필요
- 단점: URL 길이 제한 (약 2KB)
- 용도: 턴 수가 적은 배틀

**방법 2: 서버 API 경유 (권장)**
```
1. 봇 서버: 배틀 데이터를 DB/Redis에 저장 → battle_id 발급
2. 미니앱 URL: https://tgpoke.com/battle?id={battle_id}
3. 미니앱: fetch('/api/battle/{battle_id}') → JSON 수신
4. 배틀 재생 완료 후: Telegram.WebApp.sendData(result)
```
- 장점: 데이터 크기 제한 없음, 캐싱 가능
- 단점: API 엔드포인트 필요
- 용도: 모든 배틀 (기본 방식)

### 4.3 서버 측 구현 (배틀 데이터 생성)

`tournament_service.py`의 결승전 처리 로직에 다음을 추가한다.

```python
# tournament_service.py (개념 코드)

async def run_final_battle(player_a, player_b, spectators):
    """결승전 배틀 실행 + 미니앱 데이터 생성"""

    # 1. 기존 텍스트 배틀 실행 (결과 도출)
    result = await _resolve_battle(player_a.team, player_b.team)

    # 2. 미니앱용 JSON 생성
    battle_data = {
        "battleId": generate_id(),
        "type": "tournament_final",
        "my": serialize_pokemon(player_a),
        "enemy": serialize_pokemon(player_b),
        "spectators": [
            {"name": s.name, "avatar": s.avatar_url}
            for s in spectators
        ],
        "turns": result.turn_log,   # 기존 배틀 로그 변환
        "winner": result.winner
    }

    # 3. Redis에 저장 (TTL 24시간)
    await redis.set(
        f"battle:{battle_data['battleId']}",
        json.dumps(battle_data),
        ex=86400
    )

    # 4. 미니앱 버튼 전송
    keyboard = InlineKeyboardMarkup([[
        InlineKeyboardButton(
            "⚔️ 3D 배틀 관전하기",
            web_app=WebAppInfo(
                url=f"https://tgpoke.com/battle?id={battle_data['battleId']}"
            )
        )
    ]])

    await bot.send_message(
        chat_id=tournament_chat_id,
        text="🏆 결승전이 시작됩니다! 아래 버튼을 눌러 3D 배틀을 관전하세요.",
        reply_markup=keyboard
    )
```

## 6. 비즈니스 효과 분석

### 6.1 유저 인게이지먼트 증가

#### 토너먼트 참여 동기 강화

현재 토너먼트 결승전은 텍스트 기반으로 결과만 표시된다. 3D 배틀 시각화가 도입되면:

| 지표 | 현재 (텍스트) | 예상 (3D) | 증가율 |
|------|-------------|-----------|--------|
| 토너먼트 참가율 | 30~40% | 60~70% | +100% |
| 결승전 관전율 | 10~20% | 80~90% | +350% |
| 토너먼트 후 활동량 | 기본 | +30% | — |

**근거:**
- "내 포켓몬이 3D로 싸우는 것을 보고 싶다" → 결승 진출 목표가 명확해짐
- 관전 자체가 콘텐츠가 됨 → 토너먼트를 그냥 지나치지 않게 됨
- 결승에서 진 유저도 "다음엔 이기겠다"는 동기 부여

#### 수집 욕구 자극

- 3D로 표현되는 포켓몬의 시각적 매력 → "이 포켓몬을 갖고 싶다"
- 이로치(샤이니)의 특별 이펙트 → 이로치 수집 동기 극대화
- 강력한 스킬 이펙트 → "이 스킬의 포켓몬이 필요해"

#### 관중 소속감

- 토너먼트에 `ㄷ`을 눌러 참가한 유저가 관중석에 등장
- "내 이름이 관중석에 떴다!" → 참가 자체의 보상감
- 관중 수가 많을수록 웅장한 스타디움 → 대규모 참여 유도

### 6.2 바이럴 효과

#### 자연 바이럴 경로

```

     │
     ├──▶ 화면 녹화/캡처 ──▶ SNS 공유 (트위터, 인스타)
```

#### 결승전 관전 문화 형성

- 정기 토너먼트 결승전이 커뮤니티 이벤트화
- "오늘 8시 결승전 같이 보자" → 정기적 활성 시간대 형성
- 결승전 하이라이트 아카이브 → 과거 명경기 회고 문화

```javascript
// 안전한 사운드 초기화
document.getElementById('start-btn').addEventListener('click', () => {

    audioManager.init();       // AudioContext.resume()
```

#### 사운드 에셋 용량

| 방안 | 효과 | 트레이드오프 |
|------|------|------------|

| 지연 로딩 | 초기 로딩 감소 | 턴 전에 미리 버퍼링 필요 |

### 8.3 모바일 성능 리스크

#### 저사양 기기 대응

**자동 품질 감지:**
```javascript
function detectPerformanceTier() {
    const gl = canvas.getContext('webgl');
    const debugInfo = gl.getExtension('WEBGL_debug_renderer_info');
    const renderer = gl.getParameter(debugInfo.UNMASKED_RENDERER_WEBGL);

    // GPU 벤치마크 기반 티어 분류
    if (isFlagship(renderer))  return 'high';
    if (isMidRange(renderer))  return 'medium';
    return 'low';
}
```

**티어별 설정:**
| 설정 | High | Medium | Low |
|------|------|--------|-----|
| 해상도 | 1x (네이티브) | 0.75x | 0.5x |
| 파티클 수 | 500 | 250 | 100 |
| 그림자 | PCF Soft (2048) | Basic (1024) | OFF |
| 블룸 | 풀 해상도 | 1/2 해상도 | OFF |
| 관중 수 | 전체 | 50% | 20% |
| 관중 애니메이션 | 개별 | 그룹 (5명 단위) | 정적 |
| 텍스처 | 1024 | 512 | 256 |
| 안티앨리어싱 | MSAA 4x | FXAA | OFF |
| 잔디 인터랙션 | ON | 간소화 | OFF |
| 목표 FPS | 60 | 30 | 30 |

**런타임 성능 모니터링:**
```javascript
// FPS 모니터링 → 자동 품질 조정
let frameCount = 0;
let lastCheck = performance.now();

function checkPerformance() {
    frameCount++;
    const now = performance.now();
    if (now - lastCheck >= 2000) {  // 2초마다 체크
        const fps = frameCount / ((now - lastCheck) / 1000);
        if (fps < 24) downgradeQuality();  // 자동 품질 하향
        if (fps > 55 && currentTier !== 'high') upgradeQuality();
        frameCount = 0;
        lastCheck = now;
    }
}
```
