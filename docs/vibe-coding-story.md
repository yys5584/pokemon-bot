# 바이브코딩으로 라이브 서비스 만들기

> 비개발자가 Claude로 9일 만에 텔레그램 포켓몬 봇을 만들어 운영한 과정과 결과
>
> 2026-03-11 | 문유

---

## 1. 먼저 결과부터

### 만든 것

텔레그램 채팅방에서 동작하는 포켓몬 수집/배틀/거래 봇.
코인 채팅방의 체류 시간을 늘리기 위한 목적으로 기획했다.

### 숫자

| 항목 | 수치 |
|------|------|
| 개발 기간 | 9일 |
| 커밋 수 | 366개 |
| 코드량 | Python 36,000줄+ |
| DB 테이블 | 20개+ |
| 명령어 수 | 40개+ |
| 포켓몬 종류 | 251종 (1~2세대) |
| 스킬 (각 포켓몬 고유) | 251개 |
| 웹 대시보드 | tgpoke.com |
| 현재 상태 | 라이브 운영 중, 유저 수백 명 |

### 나의 배경

- Python, SQL 모름. 코드를 직접 쓸 줄 모른다.
- 기획/PM 경력. 유저 경험 설계와 우선순위 판단이 본업.
- 모든 코드는 Claude(AI)가 작성. 나는 뭘 만들지, 왜 만드는지를 결정했다.

---

## 2. 목적 — 왜 이걸 만들었나

### 문제

텔레그램 코인 채팅방은 시세가 나쁘면 조용해지고, 조용해지면 사람이 빠지고, 사람이 빠지면 방이 죽는다.

**시세와 무관하게 사람들이 매일 채팅방에 오게 하려면?**

### 가설

채팅방 안에 포켓몬 수집 게임을 넣으면, 시세와 관계없이 사람들이 온다.

- 포켓몬은 20~40대 한국 남성 대부분이 아는 IP
- "ㅊ" 한 글자로 시작 가능. 진입장벽 제로
- 수집욕이 체류 시간을 만든다

---

## 3. 게임 루프 — 뭘 만들었나

### 전체 구조

```
[입문]  스폰 → 포획 → 도감 수집
         ↓
[성장]  배틀 → 팀 구성 → 토너먼트
         ↓
[경쟁]  랭크전 → 시즌 → 칭호
         ↓
[경제]  거래소 → 교환 → 합성
         ↓
[엔드]  캠프 → 이로치 제작
```

### 단계별 설명

**입문: 수집**
- 채팅방에 랜덤으로 포켓몬이 스폰된다
- "ㅊ" 입력하면 포획 시도. 포켓볼 종류에 따라 확률이 다르다
- 251종을 모으는 도감 시스템

**성장: 배틀**
- 포켓몬 3마리로 팀을 짜고, 다른 유저와 1:1 배틀
- 각 포켓몬마다 고유 스킬 1개. 타입 상성 적용
- 토너먼트(대회), 아케이드(NPC 배틀) 제공

**경쟁: 랭크**
- 시즌제 랭크전. ELO 기반 매칭
- 시즌 종료 시 순위별 보상
- 칭호 시스템 (도감 수, 랭크 달성 등)

**경제: 거래**
- 유저 간 포켓몬 거래소 (등록 → 구매)
- 1:1 교환, 포켓몬 합성
- 게임 내 재화(포켓코인) 순환

**엔드게임: 캠프**
- 포켓몬을 구역에 배치하면 자동으로 조각(재화) 획득
- 조각을 모아 이로치(색이 다른 특별한 포켓몬)로 전환
- 채팅방 단위 캠프 레벨. 협력 콘텐츠

### 핵심 설계 원칙

모든 기능이 **채팅방 안에서** 동작한다.
다른 유저가 뭘 잡았는지, 뭘 샀는지 보여야 "나도 해볼까"가 된다.
봇 메시지는 일정 시간 후 자동 삭제 — 채팅방의 본래 용도를 방해하지 않는다.

## 4. 과정 — 어떻게 만들었나

### 개발 방식: 바이브코딩

```
나 (기획자)         Claude (AI)
────────────        ──────────
뭘 만들지 결정      코드 작성
왜 필요한지 설명    아키텍처 설계
유저 피드백 해석    버그 수정
우선순위 판단       성능 최적화
```

나는 코드를 읽지도, 쓰지도 않았다.
대신 "이런 경험을 만들어야 한다"를 구체적으로 설명했다.

### 9일간의 타임라인

```
Day 1    기본 포획 + 도감
Day 2    포켓볼 시스템, 2세대 추가
Day 3-4  배틀 시스템
Day 5    토너먼트, 아케이드
Day 6    거래소, 교환, 합성
Day 7    랭크전, 시즌, 이로치
Day 8    대시보드 리뉴얼
Day 9    스킬 효과, 타입 상성, 밸런싱
Day 10   캠프 시스템 (엔드게임)
```

### AI에게 말하는 방식

안 되는 예: "배틀 시스템 만들어줘"
→ 제네릭한 결과가 나온다.

되는 예: "코인방 20대 남성들이 서로 도발하면서 즐길 수 있는 배틀. 이기면 티배깅 버튼이 나오게 해줘"
→ 타겟 유저에 맞는 결과가 나온다.

**도메인 이해와 유저 감각이 프롬프트의 품질을 결정한다.**

### 기능 설계 패턴

모든 기능은 이 순서를 따랐다:

```
1. 유저 관찰  — "포켓몬을 다 모은 유저들이 이탈하고 있다"
2. 래퍼런스  — "네코 아츠메는 놓고 기다리는 방식으로 풀었다"
3. 현실 제약  — "코인방이라 메시지를 적게 보내야 한다"
4. 구현 요청  — Claude에게 위 맥락과 함께 요청
```

### 패치 사이클

```
유저: "배틀이 너무 오래 걸려요"
나:   (5분 후) "고쳤어, 다시 해봐"
유저: "오 빨라졌다"
```

기존 개발: 이슈 등록 → 스프린트 → 개발 → QA → 배포 (일~주 단위)
바이브코딩: 피드백 → Claude에게 설명 → 수정 → 배포 (분 단위)

유저 피드백 → 수정 → 배포까지 걸리는 시간이 5분이면,
유저는 "내 말이 반영됐다"고 느낀다. 이게 초기 리텐션을 만들었다.

## 5. 결과 — 뭘 얻었나

### 정량적 결과

- 9일 만에 라이브 서비스 출시
- 봇 도입 후 채팅방 메시지량 체감 증가
- 시세가 나빠도 포켓몬 관련 대화가 유지됨
- 유저 수백 명이 매일 사용

### 검증된 가설

"시세와 무관한 체류 이유를 만들면 채팅방이 유지된다" — 맞았다.
포켓몬 이야기는 시세 이야기보다 가볍고, 누구나 참여할 수 있다.

### 바이브코딩에서 배운 것

**1. "뭘 만들지"가 "어떻게 만들지"보다 중요하다**
AI가 코딩을 대신하면, 차별화는 기획력에서 나온다.
같은 AI를 쓰더라도 "뭘 만들어야 하는지"를 아는 사람이 다른 결과를 낸다.

**2. 맥락이 풍부한 프롬프트가 좋은 결과를 만든다**
"배틀 만들어줘"와 "코인방 20대가 도발하며 즐길 배틀"은 결과가 완전히 다르다.
도메인 이해, 유저 심리, 현실 제약을 프롬프트에 담아야 한다.

**3. 빠른 피드백 루프가 제품을 만든다**
5분 안에 유저 피드백을 반영할 수 있으면,
일주일에 수십 번의 개선이 가능하다. 이건 전통적 개발로는 어렵다.

## 6. 앞으로 — 다음 단계

- 3세대 포켓몬 추가 (386종)

- 다른 채팅방으로 확장

### 바이브코딩 활용 측면

이 경험에서 느낀 바이브코딩의 조건:

```
1. 도메인 이해가 있어야 한다
   → AI는 "코인방 유저 심리"를 모른다. 내가 알려줘야 한다.

2. 우선순위를 정할 수 있어야 한다
   → AI는 시키면 뭐든 만든다. "지금 이걸 만들어야 하는가?"는 내가 판단한다.

3. 제약 조건을 설계에 녹일 수 있어야 한다
   → "채팅방을 더럽히면 안 돼" 같은 제약을 기능으로 변환하는 건 사람의 일이다.

4. 피드백을 해석할 수 있어야 한다
   → 유저가 "재미없어"라고 하면, 뭐가 재미없는지 판단하는 건 AI가 아니라 나다.
```

바이브코딩은 "코딩을 못해도 개발할 수 있다"가 아니다.
**"기획력이 있는 사람이 기획력만으로 프로덕트를 배포할 수 있다"**는 것이다.

개발자분들은 이미 코딩 능력이 있으니,
여기에 바이브코딩을 더하면 프로토타이핑과 반복 속도가 훨씬 빨라질 수 있다.
